### F-01 · HIGH · Password recovery (MM-DEC rev02 §5, locked) is unimplemented for patients — and for providers exists only as the admin manual path — and is absent from the launch checklist _(re-verified inline)_

The locked identity strategy requires patient password recovery via WhatsApp
OTP / email / SMS and provider self-service recovery via verified email → OTP
→ SMS, with admin manual recovery reserved for "exceptional cases". What
exists is only the exceptional path: `identity.recoverProviderAccount`, a
`roleProcedure("admin")`. No reset/recovery command, token flow, or client UI
exists anywhere. ADR-0016 recorded this as a carry-in "to the next identity
slice"; no slice, ADR, or owner disposition has touched it since, and
ADR-0031's checklist — which claims to enumerate every launch blocker — omits
it. At launch, any user who forgets a password has no self-service path back
into an account holding their bookings and clinical record; a locked document
is silently unmet at go-live.

- **Evidence:** MM-DEC rev02 §5 (lines 57–61). `grep -rniE "password.?reset|forgot|recovery"` across `apps/api/src`, `packages/contracts/src` → only `recover-provider-account.ts` and its wiring; identity router read: sole recovery procedure at `apps/api/src/modules/identity/router.ts:175` is `roleProcedure("admin")`. ADR-0016:77–80 (carry-in #1); ADR-0031 read in full — no recovery item; MM-DES-003 §10 does not disposition it.
- **Owner:** Owner ruling — implement pre-launch or disposition as a dated ADR-0031 amendment. Note that accepting it for launch amends a _locked_ document's requirement, which per CLAUDE.md needs the ADR + dated-amendment pattern, not silence.

### F-02 · HIGH · No privacy policy, terms of service, or account-deletion flow exists anywhere in the repo — HG-1 store submission and lawful operation both depend on them _(re-verified inline)_

Zero privacy-policy or terms content exists in web pages, mobile screens, i18n
catalogs, or docs; no account-deletion flow exists in the identity module or
any client. Apple and Google both require a privacy policy URL and — for apps
supporting account creation — an in-app account-deletion path, so HG-1
(checklist item 1, which includes store submission) fails these requirements
as things stand. Aggravator: `docs/runbooks/data-retention-erasure.md`'s
erasure matrix for user/auth tables prescribes an "account deletion flow
(identity module)" **that does not exist**, making that documented erasure
path unexecutable. A health-data platform collecting patient PII would launch
with no published privacy terms.

- **Evidence:** `grep -rniE "privacy.?policy|terms.of.(service|use)|delete.{0,8}account|account.{0,8}deletion"` across `apps`, `packages`, `docs` → sole hit is the erasure runbook's reference to the nonexistent flow. `grep -rni 'deleteAccount|deleteUser|account.*delet'` in identity module + contracts + clients → only `deleteUserSessions` (`recover-provider-account.ts:37`). ADR-0031:36 (HG-1 includes store submission); `docs/runbooks/release-cut-mobile.md`, ADR-0019/0024: no privacy/data-safety/deletion mention.
- **Owner:** Owner + a named standalone slice (policy/terms content is owner/legal work; account deletion is an identity-module slice). Hard prerequisite of HG-1.

### F-03 · HIGH · A total API outage is undetectable by the launch observability posture: both alerts go silent when the API dies, no uptime/synthetic probes exist, and none of MM-ARC-002 §10.9's incident runbooks were written _(re-verified inline)_

Both committed launch alerts evaluate metrics exported by the API process
itself. The 5xx-rate rule is explicitly `noDataState: OK` ("no traffic ⇒ no
error rate; don't page on quiet") — a dead API emits nothing and pages nobody
— and the outbox rule is `noDataState: NoData`, which is not a firing state.
MM-ARC-002 §10.8 requires uptime probes on `/health` + `/ready` plus a
synthetic guest-booking probe; §10.9 requires per-scenario incident runbooks
(API down, DB degraded, outbox stalled, provider outage, OTP abuse, data
breach). None of this exists, and no ADR-0031 checklist item covers it beyond
HG-2's dashboard provisioning. A post-launch outage of a booking/clinical
platform proceeds undetected until a human notices, with no written response
procedure when they do.

- **Evidence:** `docs/observability/alerts/alert-rules.yaml` (both `noDataState` values read; the `OK` line carries the quoted comment). `ls docs/runbooks/` → backup-restore, data-retention-erasure, release-cut-mobile, 4× secrets-rotation — no incident scenarios. MM-ARC-002 §10.8 (line 656, synthetic probe line 658), §10.9 (lines 660–663). `grep -rni 'uptime|on-call|incident' docs/` → hits only inside MM-ARC-002 itself. ADR-0031 items 1–12 read.
- **Owner:** Owner — extend HG-2 scope (external uptime probe + minimal incident runbook set) or disposition explicitly via dated ADR-0031 amendment.

### F-05 · HIGH · The web guest-booking screen classifies booking errors by regex over `error.message` — a convention #11 violation on the flagship flow, with the typed `appCode` available and already used by mobile

`apps/web/app/[locale]/book/[slug]/page.tsx:243-246` defines
`classifyBookingError(message)` as `/conflict|taken|SLOT/i.test(message) ?
"slotTaken" : "failed"`, invoked at line 107 with `book.error.message` — while
the function's own comment claims "Typed error codes ride the message
envelope's code field". Classification currently works only because the
server's English prose happens to contain "slot" ("This slot is already
booked", `booking/shared.ts:144`). Any rewording, localization, or proxying of
server messages — all permitted, since messages are explicitly non-contractual
per convention #11 — silently downgrades the slot-taken retry affordance to
generic "failed" on the guest-booking flow, with no test or type error to
catch it; conversely any unrelated error whose message contains
"conflict"/"taken"/"slot" is misreported to the patient as slot-taken. The
typed path is proven viable in the same flow on mobile
(`apps/mobile/app/book/[slug].tsx:75` switches on
`error.data?.appCode === ErrorCode.SLOT_UNAVAILABLE`). Found independently by
two area auditors; both verifiers confirmed. Introduced by `85b8a52` (Phase 8
Slice 4) — one day after MM-QA-002's audited revision and outside MM-QA-003's
mobile scope, so absent from both baselines. It is the **sole**
message-parsing site in either client (repo-wide grep).

- **Evidence:** page.tsx:107, :243–247 read verbatim (twice, independently); `kernel/trpc.ts:19` ships `appCode` (comment line 9: "Clients switch on codes, never [messages]"); server throw sites `booking/shared.ts:121,144`, `commands/book-appointment.ts:83`; repo-wide `rg '\.message\.(includes|startsWith|match|indexOf)'` + `.message ===` sweep → this is the only parser; `git log -S classifyBookingError` → 85b8a52.
- **Owner:** Web booking screen — switch on `error.data.appCode` (pattern already on mobile). One-file fix.
